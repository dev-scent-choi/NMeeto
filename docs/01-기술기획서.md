# AI 모의면접 서비스 기술 기획서

작성일: 2026-07-20 · 버전: v0.1 (초안)

---

## 1. 제품 개요

### 1.1 목표

지원자가 특정 회사·직무에 지원했다는 정보를 입력하면, AI가 해당 회사의 실제 면접 스타일과 기출 질문, 지원자의 이력서 내용을 반영하여 **실제 면접과 유사한 대화형 모의면접**을 진행하는 서비스. 텍스트 → 음성 → 화상 순으로 몰입도를 확장한다.

### 1.2 핵심 가치 제안

기존 서비스(단순 질문 리스트 제공, 녹화 후 일방향 피드백)와의 차별점은 다음 세 가지다.

1. **맥락 맞춤**: 회사 정보 + 직무기술서(JD) + 지원자 이력서를 결합해 "그 회사가 나에게 물어볼 법한" 질문을 생성.
2. **양방향 대화**: 답변 내용을 이해하고 즉석에서 꼬리질문(follow-up)을 던지는 것. 이것이 "실제 면접 같은 느낌"의 본질.
3. **정량·정성 피드백**: 답변 구조(STAR), 발화 속도, 필러워드, 내용 적합도 등을 리포트로 제공.

### 1.3 성공 지표 (초기)

- 면접 1회 완주율 ≥ 70%
- 세션당 평균 꼬리질문 발생 수 ≥ 3회 (대화형으로 작동한다는 증거)
- 재방문율(7일) ≥ 30%
- 면접 1회당 원가 ≤ 목표 과금액의 30%

## 2. 사용자 시나리오

1. 회원가입 후 이력서(PDF/텍스트) 업로드 → 파싱되어 프로필로 저장.
2. 면접 세션 생성: 회사명, 직무명, JD 붙여넣기(또는 채용공고 URL), 면접 유형(인성/직무/종합), 난이도, 시간(10/20/30분) 선택.
3. 시스템이 질문 계획(Question Plan)을 생성: 기출 질문 검색 결과 + 이력서 기반 개인화 질문을 조합.
4. 면접 진행: AI 면접관이 인사 → 질문 → 답변 청취 → 꼬리질문 또는 다음 질문 → 마무리.
5. 종료 후 피드백 리포트 생성: 답변별 평가, 전체 총평, 개선 예시 답변, 발화 습관 분석.
6. 히스토리에서 과거 세션 다시보기, 특정 질문만 재도전.

## 3. 시스템 아키텍처

### 3.1 전체 구성

```
[클라이언트 (웹 SPA / 모바일)]
        │  HTTPS / WebSocket / WebRTC
        ▼
[API Gateway + 인증]
        │
        ├── [Core API 서버] ─── PostgreSQL (사용자, 세션, 질문, 평가)
        │                  └── Object Storage (이력서, 녹음/녹화 파일)
        │
        ├── [Interview Orchestrator]  ← 면접 상태머신 + LLM 호출
        │        ├── LLM API (질문 생성, 꼬리질문, 평가)
        │        ├── STT (음성→텍스트)
        │        └── TTS (텍스트→음성)
        │
        ├── [Question Sourcing Worker]  ← 기출 질문 수집/검색 (비동기)
        │        └── 검색 API / 크롤러 / 자체 질문 DB
        │
        └── [Feedback Worker]  ← 세션 종료 후 리포트 생성 (비동기 큐)
```

### 3.2 컴포넌트별 책임

**Core API 서버**: 회원/인증, 세션 CRUD, 이력서 업로드·파싱, 결제. 무상태(stateless)로 수평 확장 가능하게 설계.

**Interview Orchestrator**: 면접의 실시간 두뇌. 세션별 상태머신을 유지하며 "지금 어느 질문 단계인지, 꼬리질문을 몇 번 했는지, 남은 시간이 얼마인지"를 관리하고 LLM 호출을 조율한다. WebSocket으로 클라이언트와 연결. 이 컴포넌트가 사실상 제품의 핵심이며, 별도 서비스로 분리해두면 이후 실시간 음성 파이프라인 교체가 쉬워진다.

**Question Sourcing Worker**: 세션 생성 시 비동기로 동작. 회사명+직무로 웹 검색을 수행해 기출 질문·면접 후기를 수집하고, LLM으로 정제하여 질문 후보 풀을 만든다. 결과는 캐싱하여 같은 회사·직무 요청 시 재사용(비용 절감 + 응답속도).

**Feedback Worker**: 면접 종료 이벤트를 큐(예: Redis + BullMQ, 또는 SQS)로 받아 전체 대화록을 분석하고 리포트를 생성. 실시간성이 필요 없으므로 큐 기반 비동기 처리로 LLM 비용이 큰 작업을 분리한다.

### 3.3 면접 상태머신

Orchestrator 내부 상태 설계. LLM에게 전체 흐름 제어를 맡기면 시간 관리와 질문 커버리지가 불안정해지므로, **흐름은 코드가, 내용은 LLM이** 담당하는 구조를 권장한다.

```
INTRO → QUESTION(i) → LISTENING → EVALUATE_TURN
                          │              │
                          │        [꼬리질문 필요?]
                          │         yes ↓        no ↓
                          └──── FOLLOW_UP(≤2회)   NEXT or WRAP_UP → CLOSING → REPORT
```

턴 종료 시마다 경량 LLM 호출로 판단한다: 답변이 (a) 충분히 구체적인가, (b) 파고들 가치가 있는 키워드가 있는가, (c) 남은 시간 대비 진도가 적절한가. 꼬리질문은 질문당 최대 2회로 제한해 면접이 한 주제에 매몰되는 것을 방지.

## 4. LLM 설계

### 4.1 역할 분리 (모델 3개 역할)

하나의 프롬프트로 모든 걸 하려 하면 품질과 비용 모두 나빠진다. 역할별로 호출을 분리한다.

| 역할 | 시점 | 모델 등급 | 설명 |
|---|---|---|---|
| Question Planner | 세션 생성 시 1회 | 상위 모델 | 기출+이력서 기반 질문 계획 수립 (JSON 출력) |
| Interviewer | 매 턴 | 중간 모델 (지연 민감) | 면접관 페르소나로 발화 생성, 꼬리질문 |
| Turn Judge | 매 턴 | 경량 모델 | 꼬리질문 여부/진행 판단 (수십 토큰 출력) |
| Evaluator | 종료 후 1회 | 상위 모델 | 전체 대화록 기반 리포트 생성 |

### 4.2 컨텍스트 구성

Interviewer의 시스템 프롬프트에 포함할 것: 회사 요약(사업, 인재상, 최근 이슈), JD 핵심 요건, 이력서 요약(원문이 아닌 LLM으로 사전 요약한 500토큰 내외), 질문 계획, 면접관 페르소나(말투, 압박 정도), 진행 규칙(발화는 2~3문장 이내, 한 번에 한 질문만).

이력서 원문 전체를 매 턴 넣으면 토큰 비용이 급증하므로, 세션 생성 시 요약본을 만들어 캐싱한다. 프롬프트 캐싱(Anthropic/OpenAI 모두 지원)을 적용하면 시스템 프롬프트 반복 비용을 크게 줄일 수 있다.

### 4.3 페르소나와 리얼리즘

"실제 같은 느낌"을 위한 디테일: 면접관 이름·직책 부여("저는 개발팀 리드 ○○입니다"), 답변에 짧은 리액션("네, 그 부분 흥미롭네요") 후 질문, 지원자가 얼버무릴 때 자연스러운 재질문, 마지막에 "저희에게 궁금한 점 있으신가요?" 역질문 단계. 이런 규칙을 프롬프트에 명시한다.

## 5. 기출 질문 수집 전략

세 가지 소스를 계층적으로 사용한다.

1. **자체 질문 DB (장기 자산)**: 직무별 표준 질문 세트를 수동 큐레이션으로 시작. 서비스 운영 중 생성된 질문 중 좋은 것을 익명화하여 축적. 이것이 장기적 해자(moat)가 된다.
2. **웹 검색 API (기본)**: 세션 생성 시 "회사명 + 직무 + 면접 질문" 검색 → 상위 결과를 LLM으로 정제. 크롤러 직접 구축보다 법적·유지보수 리스크가 낮다.
3. **크롤링 (신중히)**: 잡플래닛·블라인드·글래스도어 등은 대부분 약관상 크롤링을 금지하며 로그인 장벽이 있다. 초기에는 피하고, 필요 시 제휴/API 계약을 검토한다. **약관 위반 크롤링은 사업 리스크이므로 아키텍처의 전제로 삼지 말 것.**

기출이 부족한 회사(중소기업 등)의 경우, 회사 홈페이지·채용공고·뉴스 검색 결과로 회사 프로필을 만들어 LLM이 질문을 생성하는 폴백 경로를 둔다.

## 6. 음성 파이프라인

### 6.1 두 가지 아키텍처 비교

**A. 캐스케이드 (STT → LLM → TTS)**

- 흐름: 클라이언트 녹음 → STT(스트리밍) → 텍스트 → LLM → TTS(스트리밍) → 재생
- 지연: 발화 종료 후 첫 음성까지 1.5~3초. 문장 단위 스트리밍 TTS로 체감 단축 가능.
- 장점: 각 단계 교체 자유, 대화록이 자연히 남음(피드백 분석에 필수), 디버깅 용이, 비용 통제 쉬움.
- 단점: 지연, 발화 종료 감지(VAD) 직접 처리 필요.

**B. 실시간 음성 API / 음성 에이전트 프레임워크**

- 예: OpenAI Realtime API, 또는 LiveKit Agents / Pipecat 위에 STT·LLM·TTS를 조립.
- 지연: 1초 미만 가능, 끼어들기(barge-in) 지원 → 대화감 우수.
- 단점: 비용 상승, 벤더 종속 또는 인프라 복잡도, 세밀한 프롬프트 제어가 캐스케이드보다 어려운 경우 있음.

**권장**: MVP는 A(캐스케이드)로 시작하되, Orchestrator와 음성 계층을 인터페이스로 분리해두고 2단계에서 LiveKit Agents 또는 Pipecat 기반 B로 전환. 면접은 일상 대화보다 턴이 길고 명확해서(질문→긴 답변) 캐스케이드의 지연이 상대적으로 덜 치명적이다.

### 6.2 세부 기술 선택지

- **STT**: OpenAI Whisper API(저렴, 한국어 준수), Deepgram(스트리밍 강점), 네이버 클로바 스피치(한국어 특화), Google STT. 한국어 인식률 벤치마크를 직접 돌려보고 결정할 것 — 후보 3개에 동일 샘플 30개를 넣어 WER 비교.
- **VAD(발화 종료 감지)**: 클라이언트에서 Silero VAD(WebAssembly 구동 가능) 또는 서버 측 스트리밍 STT의 endpointing 기능. "침묵 1.2초 = 발화 종료" 같은 단순 규칙은 생각하며 말하는 면접 특성상 오작동이 잦으므로, 침묵 시간 + 문장 완결성(경량 판단)을 조합.
- **TTS**: ElevenLabs(품질 최상, 고가), OpenAI TTS(무난), 네이버 클로바 보이스 / 타입캐스트(한국어 자연스러움 강점). 면접관 톤(차분, 신뢰감)에 맞는 보이스 선정이 중요.

### 6.3 오디오 전송

MVP: 클라이언트에서 MediaRecorder로 청크 녹음 → WebSocket으로 서버 전송. 2단계(실시간): WebRTC로 전환(LiveKit 사용 시 자동 해결). 원본 오디오는 피드백 분석(발화 속도, 필러워드)용으로 Object Storage에 보관(사용자 동의 필수).

## 7. 화상 면접

화상의 가치를 둘로 분리해서 판단한다.

**(a) 지원자 촬영 → 분석**: 실용 가치가 크다. 카메라 녹화 후 시선 처리, 표정, 자세에 대한 피드백 제공. MVP+1 단계에서 브라우저 getUserMedia 녹화 + 종료 후 비전 모델 분석(프레임 샘플링)으로 구현 가능. 실시간 분석은 불필요 — 사후 분석이면 충분하고 훨씬 저렴하다.

**(b) AI 면접관 아바타**: 몰입감 요소. 선택지는 정적 프로필 이미지 + 음성 파형 애니메이션(비용 0, 의외로 충분히 자연스러움) → 립싱크 2D 아바타 → 실시간 스트리밍 아바타(D-ID, HeyGen 등, 분당 과금이 상당함). **아바타는 3단계로 미루고, 그 전에는 이미지+파형으로 충분하다.** 어색한 립싱크는 오히려 몰입을 깬다(불쾌한 골짜기).

WebRTC 인프라는 직접 구축하지 말고 LiveKit(오픈소스 셀프호스팅 가능) 또는 Daily.co를 사용한다.

## 8. 피드백 엔진

세션 종료 후 Feedback Worker가 생성하는 리포트 구성:

1. **답변별 평가**: 질문 의도 대비 적합성, 구체성(수치·사례 포함 여부), STAR 구조 충족도, 점수(5점 척도) + 근거 + 개선 예시 답변.
2. **발화 습관 분석** (오디오 기반): 분당 어절 수(발화 속도), 필러워드 빈도("어", "음", "그", "약간" 등 — STT 전사에서 카운트), 답변 평균 길이, 침묵 패턴.
3. **총평**: 강점 2~3개, 개선점 2~3개, 다음 연습 추천 질문.
4. **직무 적합 관점**: JD 요건별로 답변에서 확인된 근거 매핑("요구역량: 대용량 트래픽 경험 → 답변에서 언급됨/미흡").

평가 일관성을 위해 루브릭(평가 기준표)을 프롬프트에 고정하고, 점수는 LLM이 근거를 먼저 쓰고 점수를 마지막에 출력하게 한다(근거 선행 방식이 점수 신뢰도를 높임).

## 9. 기술 스택 제안

| 영역 | 선택 | 근거 |
|---|---|---|
| 프론트엔드 | Next.js (React) + TypeScript | 웹 우선 전략, 오디오/WebRTC 생태계 성숙 |
| 모바일 (후순위) | React Native | 웹 코드·팀 역량 재사용 |
| 백엔드 | Node.js(NestJS) 또는 Python(FastAPI) | WebSocket·스트리밍 처리 용이. 음성/ML 도구 활용 예정이면 FastAPI 권장 |
| 실시간 통신 | WebSocket(MVP) → LiveKit(2단계) | 점진 전환 |
| DB | PostgreSQL + pgvector | 세션 데이터 + 질문 임베딩 검색을 한 DB로 |
| 큐 | Redis + BullMQ (또는 SQS) | 피드백 비동기 처리 |
| 스토리지 | S3 호환 Object Storage | 이력서·녹음·녹화 |
| LLM | Claude API / OpenAI (역할별 등급 분리) | 4.1절 참조. 추상화 계층 두고 교체 가능하게 |
| 인프라 | 초기: Vercel(FE) + 컨테이너 1~2대(BE) | 오버엔지니어링 방지. 쿠버네티스는 트래픽 검증 후 |

## 11. 보안·개인정보 (한국 기준 필수 검토)

이력서와 음성·영상은 민감한 개인정보다. 개인정보보호법 관점에서 초기부터 지켜야 할 것:

- 수집·이용 동의: 이력서, 음성 녹음, 영상 녹화 각각 명시적 동의. 특히 **음성·영상 분석은 별도 고지** 필요.
- 보관 정책: 보유 기간 명시(예: 탈퇴 시 즉시 또는 90일 후 파기), 사용자가 세션 데이터 삭제 가능하게.
- 제3자 제공: LLM/STT/TTS 외부 API로 데이터가 전송됨을 처리방침에 명시. 각 벤더의 데이터 학습 미사용(zero retention) 옵션 확인·계약.
- 저장 암호화(at-rest) + 전송 암호화(TLS), 이력서 파일 접근은 서명된 URL로만.
- LLM 프롬프트에 주민번호 등 불필요 식별정보가 들어가지 않게 이력서 파싱 단계에서 마스킹.

## 13. 개발 로드맵

**Phase 0 — 기술 검증 (1~2주)**
한국어 STT 3종 벤치마크, TTS 보이스 청취 테스트, 꼬리질문 프롬프트 프로토타입(텍스트만), 기출 검색→정제 파이프라인 손검증. *여기서 품질이 안 나오면 스택을 바꾸는 게 이후보다 100배 싸다.*

**Phase 1 — 텍스트 면접 MVP (3~5주)**
회원/이력서 업로드·파싱, 세션 생성 + Question Planner, 텍스트 채팅 면접(상태머신 + 꼬리질문), 텍스트 기반 피드백 리포트. → 이 시점에 소수 사용자 테스트로 "면접 같은가?"를 검증.

**Phase 2 — 음성 (3~4주)**
캐스케이드 파이프라인(스트리밍 STT/TTS), VAD, 발화 습관 분석 추가, 프롬프트 캐싱·비용 최적화.

**Phase 3 — 실시간성 + 화상 (4~6주)**
LiveKit 기반 전환(지연 단축, barge-in), 지원자 카메라 녹화 + 사후 시선/표정 피드백, 면접관 이미지+파형 UI 고도화.

**Phase 4 — 고도화 (지속)**
아바타 실험(A/B로 몰입도 검증 후 도입 결정), 자체 질문 DB 축적·추천, 모바일 앱, 기업 제휴(B2B 채용 연습 시장).

## 14. 주요 리스크와 대응

1. **꼬리질문 품질이 낮으면 제품이 성립하지 않음** → Phase 0에서 가장 먼저 검증. 실패 사례를 모아 프롬프트/판단 로직 반복 개선.
2. **한국어 STT 오인식** (기술 용어, 회사명) → 세션 컨텍스트의 고유명사를 STT 힌트(custom vocabulary)로 주입.
3. **지연으로 인한 몰입 붕괴** → "면접관이 메모하는" UI/사운드로 1~2초를 자연스럽게 소화, 문장 단위 TTS 스트리밍.
4. **크롤링 법적 리스크** → 검색 API + 자체 DB 중심 전략(5장).
5. **LLM 비용 폭증** → 역할별 모델 등급 분리, 프롬프트 캐싱, 질문 풀 캐싱, 세션 시간 상한.
6. **환각(존재하지 않는 회사 정보 언급)** → 회사 프로필은 검색 근거 기반으로만 작성, 근거 없으면 일반 질문으로 폴백하도록 프롬프트에 명시.
